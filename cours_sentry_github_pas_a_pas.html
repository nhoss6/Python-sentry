Sentry + GitHub Issues avec Python Flask

Support pas à pas pour expliquer à des élèves comment connecter une application Python à Sentry, provoquer une erreur volontaire, puis transformer cette erreur en ticket GitHub.

Objectif Déroulé détaillé Architecture GitHub App Sentry Dépôt GitHub Code Python Variables Test erreur Issues Chemins À dire aux élèves

1. Objectif du cours

Le but est de montrer un vrai workflow professionnel :

  1. Une application Flask tourne en local.
  2. Une route provoque volontairement une erreur.
  3. Sentry capture cette erreur avec la stack trace.
  4. Depuis Sentry, on crée ou on lie une GitHub Issue.
  5. GitHub devient l’endroit où l’équipe suit la correction.
Message clé : Sentry sert à détecter et diagnostiquer l’erreur. GitHub Issues sert à organiser le travail de correction.

2. Déroulé ultra détaillé, pas à pas

Cette section est pensée comme un script de formation. Tu peux la suivre à l’écran, étape par étape, pendant que les élèves reproduisent la manipulation.

Fil rouge à annoncer : “On part d’une application Python Flask. On provoque une erreur volontaire. Sentry la détecte. Ensuite, on transforme cette erreur en ticket GitHub pour organiser la correction.”

0

Quand une application plante, le vrai sujet n'est pas seulement l'erreur.

Étape 7 — Créer le fichier .env local

7

Commande :

copy .env.example .env

Contenu attendu dans .env :

SENTRY_DSN=votre_dsn_sentry
ENVIRONMENT=development
APP_RELEASE=api-python-mysql-swagger@1.0.0
SENTRY_TRACES_SAMPLE_RATE=0.0

Explication variable par variable :

Variable Rôle
SENTRY_DSN Adresse technique qui dit au SDK Sentry où envoyer les erreurs.
ENVIRONMENT Nom de l’environnement : development, staging, production.
APP_RELEASE Version de l’application visible dans Sentry.
SENTRY_TRACES_SAMPLE_RATE Taux de tracing. Pour cette démo, 0.0 suffit.
Règle absolue : ne jamais commiter le fichier .env. Il contient le vrai DSN Sentry.

Étape 8 — Montrer le code d’initialisation Sentry

8

Fichier à ouvrir : app.py

Code à expliquer :

load_dotenv()

sentry_dsn = os.getenv("SENTRY_DSN")
if sentry_dsn:
    sentry_sdk.init(
        dsn=sentry_dsn,
        integrations=[FlaskIntegration()],
        environment=os.getenv("ENVIRONMENT", "development"),
        release=os.getenv("APP_RELEASE", "api-python-mysql-swagger@1.0.0"),
        send_default_pii=False,
        traces_sample_rate=float(os.getenv("SENTRY_TRACES_SAMPLE_RATE", "0.0")),
    )

Explication ligne par ligne :

  • load_dotenv() charge les variables du fichier .env.
  • os.getenv("SENTRY_DSN") lit le DSN sans l’écrire dans le code.
  • if sentry_dsn: évite de planter si le DSN n’est pas encore configuré.
  • FlaskIntegration() connecte Sentry à Flask.
  • environment permet de savoir si l’erreur vient de dev ou prod.
  • release permet de relier une erreur à une version du code.
  • send_default_pii=False limite l’envoi de données personnelles.

Étape 9 — Montrer la route qui provoque l’erreur

9

Route :

http://127.0.0.1:5000/debug-sentry

Code correspondant :

@app.route("/debug-sentry")
def trigger_error():
    if os.getenv("ENVIRONMENT", "development") != "development":
        return {
            "error": "Route de demonstration desactivee hors environnement development"
        }, 403

    raise RuntimeError("Erreur volontaire pour demonstration Sentry + GitHub Issues")

Explication :

Cette route est volontairement dangereuse, mais seulement en environnement development. Elle sert à générer une exception contrôlée pour vérifier que Sentry fonctionne.

À dire : cette route ne doit pas rester accessible en production. Elle est ici pour la démonstration.

Étape 18 — Résumer le workflow complet

18
1. Le développeur lance Flask.
2. L'utilisateur ou le test appelle une route.
3. Une exception Python est levée.
4. Le SDK Sentry capture l'exception.
5. Sentry affiche l'erreur, la stack trace et le contexte.
6. Depuis Sentry, on crée une GitHub Issue.
7. L'équipe suit la correction dans GitHub.
8. Une branche et une pull request peuvent corriger le problème.

3. Architecture expliquée simplement

Application Python

L’application Flask contient les routes API et la route de test /debug-sentry.

Sentry

Sentry reçoit les exceptions, affiche la stack trace, l’environnement et la version de l’application.

GitHub Issues

GitHub reçoit le ticket de suivi pour que l’équipe puisse corriger et documenter le problème. 

Flask /debug-sentry
        ↓
Exception Python
        ↓
Sentry capture l'erreur
        ↓
Create GitHub Issue
        ↓
Ticket dans nhoss6/Python-sentry

Dans app.py, Sentry est initialisé uniquement si SENTRY_DSN existe.

sentry_dsn = os.getenv("SENTRY_DSN")
if sentry_dsn:
    sentry_sdk.init(
        dsn=sentry_dsn,
        integrations=[FlaskIntegration()],
        environment=os.getenv("ENVIRONMENT", "development"),
        release=os.getenv("APP_RELEASE", "api-python-mysql-swagger@1.0.0"),

À ne pas faire : ne jamais écrire le DSN Sentry directement dans le code source. Même si le dépôt est privé, un secret ne doit pas être versionné.

12. Script d’explication pour les élèves

Phrase d’introduction

“Aujourd’hui, on ne va pas seulement voir une erreur dans le terminal. On va mettre en place un vrai circuit professionnel : l’erreur part de l’application, arrive dans Sentry, puis devient une GitHub Issue.”

Phrase sécurité

“Un DSN Sentry est une information sensible. On ne l’écrit jamais dans le code. On utilise une variable d’environnement, ici SENTRY_DSN.”

Phrase DevOps

“Le but de l’observabilité n’est pas seulement de voir qu’une erreur existe. Le but est de réduire le temps entre la détection, la compréhension et la correction.”

Phrase de conclusion

“Sentry répond à la question : qu’est-ce qui a cassé ? GitHub Issues répond à la question : qui corrige, quand, et comment suit-on le travail ?”
